Incremental Economics

Overview

While the previous workflow for performing incremental Economic calculations remain in esi.manage, an additional, more prescriptive approach has been included. The Incremental Ring Fence is part of the Group Behaviour within a Budget – which means that if it has been configured for that budget, when you perform any Economic Analysis against those projects, the Incremental Ring Fence calculation approach will be applied.

This calculation approach (and the results) are deltas:

For any project (project n+20), what will be reported in the Economics Analysis results are:

Aggregate Cashflow (Base + project 1 through 20) – Aggregate Cashflow (Base + project 1 through 19).

Projects that are part of a Ring Fence will either be identified as Base, or Incremental Projects. For those that are Incremental, you will set a Dependency Order based on your criteria; that controls the order of the calculations.

The Dependency Order, once it has been set for a Project, is visible on the Project Detail page:

Note: when Projects that are part of an Incremental Ring Fence are copied to another Budget, their Dependency Order is reset, and if a Ring Fence existed in that Budget that the project will belong to, it will be part of the Base. You will need to move it to the Incremental Projects list in the Target Budget, and set an appropriate Dependency Order.

Creating an Incremental Ring Fence Group

Within the Projects Perspective, select the Groups page tab next to the Project List and choose Add a new Group:

 

The Type will be Incremental Ring Fence – the Budget will be defaulted to the Budget you are in.

Choose the Fiscal Model Node (based on the Fiscal Model Configuration) that defines the Group. A Manager must be assigned that provides security control over the ability to modify the Group.

The Group Screen for building the Incremental Order will start with all projects being identified as Base. Select the projects and use the Right Arrow to move them across into the Incremental Projects table.

Note, there can be multiple Projects that represent the Base – however, they will be evaluated together and presented as a single line in the Economics Results.

The Incremental Projects Table can be copied from and pasted into, if you have another source that can define the Dependency Order. It is a good practice to leave gaps in the order for the initial configuration, as it provides the ability to move projects up or down in the list. The calculation order (regardless of any sorting of the table) will be ascending in the Dependency order.

Running an Incremental Analysis

When the Incremental Ring Fence exists in the Budget, running an Economic Analysis against a subset of the projects that are in the Ring Fence will “potentially” result in other projects being calculated as well. A project’s presence in an Incremental Ring Fence will result in Incremental calculations being performed. Since this is a Delta calculation, there must be “previous” project(s) calculated to perform the delta against.

The Analysis Mode is Simple. The progress will take longer than running the projects without the Incremental Ring Fence, as the delta must be determined.

Once calculations are complete, the results can be view in the Analysis Perspective.

The order that the projects are displayed in will be the Dependency order. The results, for Pad 0029 for example, is the Delta between “Base, Pad 0015, Pad 0019 and Pad 0029” and “Base, Pad 0015 and Pad 0019”. This approach is designed to indicate a development order (projects with better incrementals should likely go first), or potentially projects that don’t increase the overall value of the portfolio.
