Distinct and Merge

The Distinct and Merge relationship is used to funnel data into a single project / attribute / node. This is useful for situations where information may be repeated, but the desired outcome is to execute the mappings only for each unique attribute. For example the following Capex table is displayed.

The desired outcome of this data would be that the Capex items on lines 2 through 4 be loaded into Example A and Capex items on lines 5 through 7 be loaded into Example B.

This is achieved by setting up a distinct node in the EDL for Project Name, then merging the Capex information into those distinct project names. Without using distinct and merge, the EDL would try to create three projects named Example A and three projects named Example B, each with one line of Capex.

The three key pieces of information highlighted are:

  • Project Name Distinct: The commands array has a repetition defined and a Distinct node called Project Name Distinct has been added which is a cell reference to A2. This is storing the distinct values which can then be used within an attribute. The attribute Project Name then references the distinct using =ESI.VAR("Project Name Distinct") so that only two projects are created.

  • Merge: The Merge node marks the starting point of mappings that will follow the distinct functionality. The merge properties references the distinct values column.

  • Capex Repeat: The Capex array is above the merge, so a repeat node is needed to be nested under the merge. The logic to load the Capex can then be defined and will execute first, followed by being rolled up into the distinct projects. The Amounts array will then repeat relative to the Capex Repeat and not the Capex array.

For information on the rest of the mappings, please see the Repeats and Repetitions section of the Help File.
